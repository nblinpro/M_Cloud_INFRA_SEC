Étude de Cas : Projet SHIELD-GATE

Sécurisation d'une infrastructure d'entreprise contre les attaques DoS et déploiement d'un accès distant sécurisé.

TechnologieOPNsense
RéseauSegmentation DMZ
AccèsWireguard + MFA
ProxyHAProxy

Architecture du Réseau

Le projet repose sur la mise en place d'un pare-feu de nouvelle génération (NGFW) OPNsense isolant les flux critiques via trois zones distinctes.

Déploiement & Reverse Proxy

Installation d'OPNsense

  • Configuration des interfaces : WAN (DHCP), LAN (10.10.1.1/24), DMZ (172.16.1.1/24).
  • Durcissement : Désactivation du port 80, changement des credentials, restriction de l'accès GUI au LAN uniquement.

Configuration de HAProxy

Pour protéger le serveur Web en DMZ, HAProxy a été configuré comme suit :

# Exemple de configuration Frontend
Frontend: HTTPS_Frontend
Listen: WAN_IP:443
SSL: Certificate_Wildcard
Action: Use Backend WEB_SRV_DMZ if { domain_logicloud.fr }

VPN Wireguard & Authentification Forte

L'accès distant est sécurisé par le protocole Wireguard, couplé à une authentification Double Facteur (TOTP).

  • Wireguard : Création des paires de clés (Pub/Priv) et configuration des AllowedIPs pour restreindre l'accès au réseau LAN.
  • MFA : Intégration de Google Authenticator via le module os-otp d'OPNsense.
  • Validation : Connexion impossible sans le jeton temporaire, même avec le certificat client.

Matrice de Filtrage (ACL)

Application de la politique Default Deny. Seuls les flux explicitement nécessaires sont autorisés :

Source Dest. Port Action
LAN WAN 80, 443, 53, 123 ALLOW
WAN DMZ 443 (via Proxy) ALLOW
LAN DMZ 22 (SSH) ALLOW
DMZ LAN ANY REJECT
